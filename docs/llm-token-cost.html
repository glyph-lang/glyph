LLM Token Cost:
How Much Do Languages Cost to Generate?

An experiment comparing the token cost of LLM-generated code across 8 programming languages, including Glyph — an experimental language designed for readability and token efficiency.

March 7, 2026 · Glyph Language Project

Introduction

When LLMs generate code, they consume tokens — both reading the prompt (input) and producing the code (output). Different programming languages require different amounts of ceremony, boilerplate, and structural overhead, which directly affects the token cost of generation.

Glyph is an experimental compiled language with Rust-inspired syntax but dramatically simpler rules: no lifetimes, no macros, optional semicolons, and built-in Result<T,E> error handling. We hypothesized that its clean, minimal syntax would make it cheap for LLMs to generate.

The results show that human readability is Glyph's dominant trait. The generated code is remarkably clean and concise, but Glyph requires a ~1,079-token language description in each prompt (since LLMs don't know Glyph natively). After adjusting for this one-time cost, Glyph is competitive with established languages — and excels at tasks that reward its type system, like error handling.

Token Counting

We measure total tokens (input + output) as reported by the OpenAI API. For Glyph, we compute adjusted tokens by subtracting the 1,079-token language description from the input count, since this is a fixed one-time cost that amortizes across many API calls in practice. All rankings use adjusted tokens to give a fair per-task comparison.

Generated code was validated by attempting compilation with each language's compiler.

Code Comparison: Error Handling

The same task — "write a function that safely divides two integers, returning an error for division by zero" — produces dramatically different code volumes across languages. Here is the actual LLM-generated output:

Glyph 6 lines · 56 output tokens 
fn safe_divide(numerator: i32, denominator: i32) -> Result<i32, String> {
  if denominator == 0 {
    ret Err(String::from_str("division by zero"))
  }
  Ok(numerator / denominator)
}
Java 57 lines · 385 output tokens 
public class SafeDivide {
  public static Result<Integer, String>
      safe_divide(int n, int d) {
    if (d == 0)
      return Result.err("division by zero");
    return Result.ok(n / d);
  }

  // Java has no built-in Result type.
  // The LLM must generate this entire
  // generic class from scratch:
  public static final class Result<T, E> {
    private final T ok;
    private final E err;
    private final boolean isOk;

    private Result(T ok, E err,
        boolean isOk) { ... }

    public static <T,E> Result<T,E>
        ok(T value) { ... }
    public static <T,E> Result<T,E>
        err(E error) { ... }

    public boolean isOk() { ... }
    public boolean isErr() { ... }
    public T unwrap() { ... }
    public E unwrapErr() { ... }

    @Override
    public String toString() { ... }
  }

  public static void main(String[] args) {
    // demo calls...
  }
}

The core logic is identical — a null check and a division — but Java requires 7x more output tokens because it lacks a built-in Result type. The LLM must generate an entire generic class with constructors, accessors, and toString. Glyph's built-in Result<T,E> eliminates this boilerplate entirely.

Discussion

Readability Is the Dominant Trait

The most striking result isn't the token count — it's how readable the generated Glyph code is. With optional semicolons, implicit returns, and minimal ceremony, the output is clean and concise. A 6-line function with clear intent, compared to 57 lines of Java boilerplate for the same task.

The Description Overhead

Glyph's main disadvantage is the ~1,079-token language description that must be included in every prompt. For established languages, the LLM already knows the syntax; for Glyph, we must teach it each time. This cost is fixed and amortizable: in a real coding assistant session, the description is sent once and reused across many code generation requests. As tasks grow more complex (producing more output tokens), the description's share of total cost decreases.

Where Glyph Excels

Glyph performs best on tasks where its type system provides built-in abstractions that other languages lack:

  • Error handling — built-in Result<T,E> saves 100–270 tokens vs. C++/Java/C
  • Struct methods — in-body method syntax avoids separate impl blocks, saving 50–80 tokens vs. Rust
  • Fibonacci — minimal syntax overhead ties Rust at 204 tokens, behind only Python (197)

Where Glyph Falls Short

Tasks requiring rich standard library support expose Glyph's youth:

  • Word frequency (686 tokens) — complex string processing and sorting pushes costs higher. Still, Glyph beats C (1,025) and is comparable to C++ (682).
  • File I/O (320 tokens) — while the adjusted cost is reasonable, compilation success depends on the LLM using the correct low-level API rather than inventing a high-level one.

Comparison to Established Languages

Python and Clojure lead overall because they are dynamically typed and highly expressive, producing very short output. Rust places 3rd thanks to its concise syntax and the models' deep familiarity with it. Glyph (4th) outperforms Go, C++, Java, and C despite the description overhead. Among statically-typed compiled languages, Glyph trails only Rust.

Conclusions

Glyph is competitive on token cost and produces remarkably readable generated code. It is not the cheapest language overall — that distinction goes to Python and Clojure — but it demonstrates that a language designed for LLM generation can match or beat established compiled languages even with the overhead of teaching the LLM its syntax from scratch.

The error handling results validate the core hypothesis: when a language provides the right built-in abstractions, LLMs produce less code, use fewer tokens, and generate more correct programs.

As Glyph's standard library matures and LLMs potentially learn its syntax natively, the description overhead will shrink or disappear, likely pushing Glyph higher in the rankings.
