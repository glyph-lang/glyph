LLM Token Cost:
How Much Do Languages Cost to Generate?

An experiment comparing the token cost of LLM-generated code across 8 programming languages — Python, Clojure, Rust, Go, C++, Java, C, and the experimental language Glyph.

March 7, 2026

Introduction

When LLMs generate code, they consume tokens — both reading the prompt (input) and producing the code (output). Different programming languages require different amounts of ceremony, boilerplate, and structural overhead, which directly affects the token cost of generation. We set out to measure these differences across a range of languages and tasks.

The comparison includes seven established languages (Python, Clojure, Rust, Go, C++, Java, C) and one experimental language, Glyph, which has Rust-inspired syntax but simpler rules (no lifetimes, no macros, optional semicolons, built-in Result<T,E>). Since LLMs don't know Glyph natively, its prompts include a ~1,079-token language description — making it an interesting case study in how an unknown language competes when the model must be taught its syntax from scratch.

The results reveal substantial differences between languages: Python and Clojure are cheapest overall, high-ceremony languages like Java and C are most expensive, and built-in abstractions (like Result types) can cut error-handling token costs in half.

Token Counting

We measure total tokens (input + output) as reported by the OpenAI API. For Glyph, we compute adjusted tokens by subtracting the 1,079-token language description from the input count, since this is a fixed one-time cost that amortizes across many API calls in practice. All rankings use adjusted tokens to give a fair per-task comparison.

Generated code was validated by attempting compilation with each language's compiler.

Code Comparison: Error Handling

The same task — "write a function that safely divides two integers, returning an error for division by zero" — produces dramatically different code volumes across languages. Here is the actual LLM-generated output:

Glyph 6 lines · 56 output tokens 
fn safe_divide(numerator: i32, denominator: i32) -> Result<i32, String> {
  if denominator == 0 {
    ret Err(String::from_str("division by zero"))
  }
  Ok(numerator / denominator)
}
Java 57 lines · 385 output tokens 
public class SafeDivide {
  public static Result<Integer, String>
      safe_divide(int n, int d) {
    if (d == 0)
      return Result.err("division by zero");
    return Result.ok(n / d);
  }

  // Java has no built-in Result type.
  // The LLM must generate this entire
  // generic class from scratch:
  public static final class Result<T, E> {
    private final T ok;
    private final E err;
    private final boolean isOk;

    private Result(T ok, E err,
        boolean isOk) { ... }

    public static <T,E> Result<T,E>
        ok(T value) { ... }
    public static <T,E> Result<T,E>
        err(E error) { ... }

    public boolean isOk() { ... }
    public boolean isErr() { ... }
    public T unwrap() { ... }
    public E unwrapErr() { ... }

    @Override
    public String toString() { ... }
  }

  public static void main(String[] args) {
    // demo calls...
  }
}

The core logic is identical — a null check and a division — but Java requires 7x more output tokens because it lacks a built-in Result type. The LLM must generate an entire generic class with constructors, accessors, and toString. Languages with built-in Result<T,E> (Glyph, Rust, Clojure) avoid this entirely.

Discussion

What Drives Token Cost?

Two factors dominate: boilerplate and standard library coverage. Languages that require class definitions, explicit access modifiers, or separate type declarations (Java, C++) consistently produce more output tokens. Languages with rich standard libraries (Python, Rust) avoid the LLM needing to implement common data structures from scratch.

Built-in Abstractions Matter

The error handling task shows this most clearly. Languages with a built-in Result type (Clojure, Glyph, Rust) cluster at 259–265 tokens, while Java requires 533 tokens — more than double — because the LLM must generate an entire generic class. This suggests that language-level abstractions for common patterns have a direct impact on LLM generation efficiency.

Dynamic vs. Static Typing

Python and Clojure lead overall, likely because dynamic typing eliminates type annotations, generic parameters, and explicit type conversions from the output. Among statically-typed compiled languages, Rust is cheapest thanks to concise syntax and deep model familiarity, followed by Glyph and Go.

The Unknown-Language Penalty

Glyph is the only language the models don't know natively, requiring a ~1,079-token language description in every prompt. Despite this overhead, it places 4th overall after adjustment — suggesting that a sufficiently minimal syntax can partly compensate for the description cost. This cost is also amortizable: in a coding assistant session, the description is sent once and reused across many requests.

Conclusions

Token cost varies substantially across languages, driven primarily by boilerplate requirements and standard library coverage. Dynamic languages (Python, Clojure) are cheapest overall. Among compiled languages, Rust leads thanks to concise syntax and deep model training. High-ceremony languages (Java, C) are consistently the most expensive.

The strongest single finding is the impact of built-in abstractions: languages with a Result type cut error-handling costs by 40–50% compared to languages without one. This suggests that language design choices directly affect LLM generation efficiency.

As a side study, Glyph demonstrates that an unknown language can compete with established ones on token cost, provided its syntax is sufficiently minimal. Its generated code is notably readable, though the ~1,079-token description overhead keeps it from topping the overall ranking.
